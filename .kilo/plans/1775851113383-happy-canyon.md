# Pipeline Architecture Improvement Plan — Scalability & Maintainability

## Context

Two independent audits (competitor-analysis.md, arch_audit.md) converge on the same critical weaknesses in CrawlerAI's orchestration layer. The extraction engine (FieldDecisionEngine, JSON-LD, `__NEXT_DATA__` interception) is strong. The problems are structural: monolithic orchestration, synchronous CPU work on the async loop, fixed concurrency, transaction tearing, and tightly coupled acquisition. This plan addresses only scalability and maintainability — no feature work, no extraction logic changes.

## Invariant Constraints

All changes must preserve the 23 architecture invariants in CLAUDE.md. Specifically:
- First-match-wins extraction hierarchy (invariant 15)
- User-owned crawl controls never rewritten (invariant 13)
- Pipeline config remains single source of truth (invariant 7)
- Typed pipeline boundaries (invariant 21)
- No reintroduction of deleted subsystems (invariant 17)

## Phase 1: Event-Loop Safety & CPU Offloading (Low Risk, High Impact)

**Problem**: `BeautifulSoup(html, "html.parser")` runs synchronously on the async event loop (`core.py`). At 8 concurrent URLs, a 4MB DOM blocks the loop for >1s, causing health check failures and dropped I/O. The arch audit scores this as the #1 critical issue.

### 1.1 — Offload DOM parsing to thread pool

**Files**: `pipeline/core.py` (lines where `BeautifulSoup` is called), `extract/service.py`, `extract/listing_extractor.py`

**Approach**:
- Wrap all `BeautifulSoup(html, parser)` calls in `await asyncio.to_thread(...)`.
- Create a shared `_parse_html(html: str) -> BeautifulSoup` utility in `pipeline/utils.py` that is always called via `to_thread`.
- This prevents any direct synchronous DOM parse on the event loop.

**Acceptance**: No `BeautifulSoup()` call exists outside `asyncio.to_thread` in any async function. Event loop latency under concurrent load stays < 50ms.

### 1.2 — Eliminate redundant DOM parsing

**Files**: `pipeline/core.py` (lines ~630-680 per arch audit), `extract/service.py`

**Approach**:
- Parse HTML into BeautifulSoup **once** per URL at the top of the extraction phase.
- Pass the parsed `soup` object downstream to all extractors, adapters, and semantic extractors instead of re-parsing raw HTML.
- This requires updating function signatures in `listing_extractor.py`, `service.py`, and `semantic_detail_extractor.py` to accept `BeautifulSoup | None` alongside raw `html`.

**Acceptance**: `BeautifulSoup()` is called exactly once per URL per pipeline execution.

## Phase 2: Pipeline Decomposition — Hook Architecture (High Risk, High Impact)

**Problem**: `_process_single_url` is 332 lines with inline I/O, CPU work, DB mutations, and retry logic. Cyclomatic complexity makes unit testing require 5+ simultaneous mocks. The arch audit scores SRP at 3/10.

### 2.1 — Define pipeline stage protocol

**New file**: `pipeline/stages.py`

**Approach**:
- Define a `PipelineStage` protocol with a single `async execute(ctx: PipelineContext) -> PipelineContext` method.
- Define `PipelineContext` dataclass holding all inter-stage state: `url`, `run`, `config`, `acquisition_result`, `soup`, `records`, `verdict`, `metrics`, `diagnostics`.
- Each stage is a standalone async callable that reads from context, does one thing, and writes results back to context.

**Stages** (mapped from current inline code in `_process_single_url`):
1. `AcquireStage` — calls `acquire()`, populates `ctx.acquisition_result`
2. `SurfaceValidationStage` — optional listing surface auto-detection
3. `BlockedDetectionStage` — runs `detect_blocked_page()`, handles adapter recovery
4. `ParseStage` — single `BeautifulSoup` parse via thread pool
5. `ExtractStage` — routes to JSON/listing/detail extraction
6. `PersistStage` — DB record creation and flush
7. `DiagnosticsStage` — artifact writing, trace building

### 2.2 — Pipeline runner

**File**: `pipeline/runner.py`

**Approach**:
- `PipelineRunner` takes an ordered list of `PipelineStage` instances and executes them sequentially.
- Pre/post hooks (`on_before_stage`, `on_after_stage`) allow injection of cross-cutting concerns (logging, timing, checkpointing) without modifying stage code.
- `_process_single_url` becomes a thin orchestrator (~30-40 lines) that:
  1. Builds `PipelineContext` from arguments
  2. Assembles stage list based on `content_type` and `surface`
  3. Calls `runner.execute(stages, ctx)`
  4. Returns `URLProcessingResult` from `ctx`

### 2.3 — Browser retry as stage composition, not inline branching

**Approach**:
- The current listing-browser-retry (lines 424–464 in core.py) is embedded as nested `if/else` inside `_process_single_url`.
- Refactor: `ExtractStage` returns a verdict. If verdict is `LISTING_FAILED` and method was `curl_cffi`, the runner re-queues `AcquireStage` + `ExtractStage` with `browser_first=True`. This is a stage-level retry policy, not inline branching.

**Acceptance**:
- `_process_single_url` is under 50 lines.
- Each stage is independently unit-testable with a single mock (its input context).
- All existing `pytest` tests pass without logic changes (only import path updates).

## Phase 3: Typed Inter-Module Contracts (Low Risk, Medium Impact)

**Problem**: `types.py` has only 2 dataclasses. Most inter-module data flows through raw `dict`s, making refactoring error-prone. Scrapling's `msgspec` approach prevents malformed kwargs from crashing deep in the stack.

### 3.1 — Expand typed boundaries

**File**: `pipeline/types.py`

**New types**:
- `PipelineContext` — replaces loose positional/keyword arguments across stage boundaries
- `ExtractionResult` — typed container for `records: list[dict]`, `verdict: str`, `winning_sources: dict`, `field_coverage: dict`
- `AcquisitionMetrics` — typed replacement for the `url_metrics` raw dict

### 3.2 — Replace raw dict returns in extraction layer

**Files**: `extract/service.py`, `extract/listing_extractor.py`, `extract/json_extractor.py`

**Approach**:
- Extraction functions currently return `list[dict]` or `(list[dict], str)` tuples.
- Convert to returning `ExtractionResult` dataclass, following the pattern already established by `URLProcessingResult`.
- Maintain tuple destructuring backward compatibility via `__iter__` as done in `URLProcessingResult`.

**Acceptance**: No raw `dict` or `tuple` crosses a module boundary in the pipeline hot path. `mypy --strict` passes on `pipeline/` package.

## Phase 4: Intra-Run URL Parallelism (Medium Risk, High Impact)

**Problem**: `_batch_runtime.py` processes URLs sequentially in a `for` loop. The global semaphore limits cross-run parallelism but provides zero intra-run concurrency. A batch of 100 URLs takes 100x single-URL time.

### 4.1 — Concurrent URL processing within a run

**File**: `_batch_runtime.py` (lines 415–482)

**Approach**:
- Replace the sequential `for url in pending_items` loop with `asyncio.gather()` or a bounded task group.
- Use a per-run semaphore (configurable, default 4) to limit concurrency within a single run.
- The existing global semaphore remains as the cross-run ceiling.
- Each concurrent task calls `_process_single_url` independently.
- Per-URL progress updates remain atomic via `_retry_run_update` with `FOR UPDATE` locks.

**Concern**: DB session sharing. Currently one session spans the entire run. With parallel URLs we need either:
- **Option A**: Per-URL session (cleaner isolation, requires refactoring `_process_single_url` to accept a session factory instead of a session).
- **Option B**: Session-per-commit with explicit savepoints (complex, risk of deadlocks).

**Recommended**: Option A — per-URL session factory. Each concurrent URL task opens its own session, commits independently, and closes cleanly. This also fixes the "long-lived session identity map bloat" problem identified in the codebase exploration.

### 4.2 — Interruptible parallel batch with checkpoint consistency

**Approach**:
- `asyncio.gather()` with `return_exceptions=True` — individual URL failures don't cancel the batch.
- On pause/kill signal, cancel pending tasks and drain in-flight tasks to completion or timeout.
- Checkpoint progress is per-URL (already implemented via `BatchRunProgressState`), so parallel completion order doesn't affect resume correctness.

**Acceptance**:
- A batch of N URLs with concurrency C completes in ~N/C * single-URL-time.
- Killing a worker mid-batch and resuming produces 0 duplicate records.
- Global semaphore still enforced across concurrent runs.

## Phase 5: Memory-Adaptive Concurrency (Low Risk, Medium Impact)

**Problem**: Fixed `URL_BATCH_CONCURRENCY` of 8. If all 8 concurrent tasks hit massive SPAs, the worker OOMs. `psutil` memory check is a hardcoded `< 500MB` threshold.

### 5.1 — Dynamic concurrency token bucket

**New file**: `services/resource_monitor.py`

**Approach**:
- Implement a `MemoryAdaptiveSemaphore` that wraps `asyncio.Semaphore`.
- Before granting a token, check `psutil.virtual_memory().percent`.
- If memory pressure > threshold (configurable via `pipeline_tuning.json`, default 85%), block until pressure drops.
- If pressure > critical threshold (default 95%), actively reduce concurrency by not releasing tokens until memory stabilizes.
- Expose `resource_snapshot()` for observability (current memory, active tokens, throttle state).

### 5.2 — Wire into batch runtime

**File**: `_batch_runtime.py`

**Approach**:
- Replace the static `_global_url_semaphore` with `MemoryAdaptiveSemaphore`.
- The per-run semaphore (from Phase 4) also delegates to the memory monitor.
- Add memory pressure to run diagnostics and `source_trace.acquisition` for post-mortem analysis.

**Acceptance**:
- Under memory pressure, concurrency automatically reduces without crashing.
- No hardcoded memory thresholds in service code (all in `pipeline_tuning.json`).
- Worker survives a batch of 500 heavy SPA pages (slower, not crashed).

## Phase 6: Transactional Checkpointing (High Risk, High Impact)

**Problem**: Record insertion (`session.flush()`) and batch progress update (`_retry_run_update` with `commit()`) are separate transactions. Worker crash between them causes phantom progress or orphaned records.

### 6.1 — Atomic per-URL unit of work

**Files**: `_batch_runtime.py`, `pipeline/core.py` (persist stage)

**Approach**:
- With per-URL sessions (from Phase 4.1 Option A), each URL's work is a single transaction:
  1. Begin transaction
  2. Insert `CrawlRecord(s)`
  3. Update `BatchRunProgressState` (same session)
  4. Commit
- If commit fails or worker dies, both record and progress are rolled back atomically.
- The `_retry_run_update` pattern with `FOR UPDATE` stays, but operates within the per-URL session instead of the run-level session.

### 6.2 — Idempotent resume with database-level dedup

**Files**: `pipeline/core.py` (`_save_listing_records`)

**Approach**:
- Add a `url_identity_key` (hash of URL + field values) to `CrawlRecord`.
- On insert, use `INSERT ... ON CONFLICT (run_id, url_identity_key) DO NOTHING` (Postgres) to prevent duplicate records on resume.
- This addresses the arch audit's memory-only deduplication criticism.

**Acceptance**:
- Hard-killing a worker mid-batch and resuming produces exactly 0 duplicate records and perfectly synced URL counts.
- No phantom progress entries exist without corresponding records.

## Phase 7: Acquisition Middleware Chain (Medium Risk, Medium Impact)

**Problem**: `acquirer.py` is 2,387 lines. The waterfall (`curl_cffi` → Playwright) is hardcoded. Adding a third-party scraping API (Zyte, BrightData) requires rewriting the core acquisition function.

### 7.1 — Acquisition strategy protocol

**New file**: `services/acquisition/strategies.py`

**Approach**:
- Define `AcquisitionStrategy` protocol: `async def acquire(request: AcquisitionRequest) -> AcquisitionResult | None`.
- Implement existing waterfall as composed strategies:
  - `HttpStrategy` — wraps current `_try_http()` (curl_cffi)
  - `BrowserStrategy` — wraps current `_try_browser()` (Playwright)
  - `AdapterRecoveryStrategy` — wraps `try_blocked_adapter_recovery()`
- `AcquisitionChain` composes strategies in order, passing to the next on failure.
- The chain is configured per-run based on domain policy, host memory, platform family.

### 7.2 — Extract current waterfall into strategies

**File**: `acquirer.py`

**Approach**:
- `_acquire_once()` (470 lines) is decomposed: each strategy handles its own success/failure logic.
- The `_needs_browser()` decision function becomes a middleware between HTTP and Browser strategies.
- `acquire()` orchestrates proxy rotation and calls `AcquisitionChain.execute()` per proxy.

**Acceptance**:
- Adding a new acquisition backend (e.g., BrightData API) requires implementing one `AcquisitionStrategy` class and registering it in the chain — no changes to `acquire()` or `_acquire_once()`.
- Existing curl → Playwright waterfall behavior is preserved exactly.

## Phase 8: Browser Pool Hardening (Low Risk, Medium Impact)

**Problem**: Browser pool uses a flat dict with psutil-based cleanup. No per-browser tab limits. Zombie browsers accumulate if psutil is missing.

### 8.1 — Mandatory psutil dependency

**File**: `requirements.txt` / `pyproject.toml`

**Approach**: Make `psutil` a hard dependency, not optional. Remove the `try/except ImportError` guard in `browser_client.py`.

### 8.2 — Per-browser context limits

**File**: `browser_client.py`

**Approach**:
- Track active context count per `_PooledBrowser`.
- Enforce `max_contexts_per_browser` (configurable via `pipeline_tuning.json`, default 4).
- When limit reached, launch a new browser instance or queue until a context frees.
- This prevents a single Playwright instance from ballooning in RAM.

### 8.3 — Structured pool with LRU eviction and health probing

**Approach**:
- Replace `dict[str, _PooledBrowser]` with a proper pool class (`BrowserPool`) that encapsulates: LRU ordering, health probing (is the browser websocket alive?), context counting, and eviction policy.
- The healthcheck loop calls `BrowserPool.evict_stale()` instead of reaching into the raw dict.

**Acceptance**:
- No zombie browser processes after 1 hour of continuous crawling.
- Memory per browser instance stays bounded by context limit.
- `psutil` absence causes a startup error, not a silent degradation.

## Implementation Order & Risk Summary

| Phase | Risk | Impact | Estimated Effort | Dependencies |
|-------|------|--------|-----------------|-------------|
| 1. CPU Offloading | Low | High | 1-2 days | None |
| 2. Pipeline Decomposition | High | High | 3-5 days | None (but Phase 1 simplifies it) |
| 3. Typed Contracts | Low | Medium | 1-2 days | Phase 2 (new types support new stages) |
| 4. URL Parallelism | Medium | High | 2-3 days | Phase 2 (per-URL session isolation) |
| 5. Memory-Adaptive Concurrency | Low | Medium | 1 day | Phase 4 (replaces static semaphore) |
| 6. Transactional Checkpointing | High | High | 2-3 days | Phase 4 (per-URL sessions) |
| 7. Acquisition Middleware | Medium | Medium | 2-3 days | Phase 2 (AcquireStage consumes strategies) |
| 8. Browser Pool Hardening | Low | Medium | 1-2 days | None |

**Total estimated effort**: 13-21 days

**Recommended execution order**: 1 → 8 → 2 → 3 → 4 → 5 → 6 → 7

Rationale: Phase 1 (CPU offloading) is a quick win with zero architectural risk. Phase 8 (browser pool) is also low-risk and standalone. Phase 2 (decomposition) is the foundation for Phases 3-7. Phases 4-6 form a dependency chain (parallelism → memory management → checkpointing). Phase 7 (acquisition middleware) is independent once the stage architecture exists.

## Out of Scope (Explicitly Deferred)

- **Session-Proxy-Fingerprint affinity** (competitor analysis Gap 1): Important for anti-bot effectiveness but is a feature concern, not scalability/maintainability.
- **OpenTelemetry tracing**: Nice-to-have observability upgrade but existing `source_trace` is functional.
- **Postgres trigger migration** (arch audit item 5): DB portability concern, not pipeline architecture.
- **Adapter HTTP bypass** (arch audit §11): Adapters doing direct HTTP instead of using centralized `http_client.py`. Important but adapter-specific, not pipeline architecture.
- **ReDoS in extraction_rules.py** (arch audit §9): Security hardening, not scalability.
